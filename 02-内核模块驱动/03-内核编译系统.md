# 内核编译系统 — Kconfig / Makefile

## 概述

Linux 内核使用 Kconfig + Makefile 构建系统来管理数万个源文件、数千个配置选项和数百种架构。

> 高通工作场景：你需要为 Qualcomm 平台添加新驱动配置选项，修改 defconfig，理解 baseline 升级中的编译问题。

## Kconfig — 内核配置系统

Kconfig 定义了内核的配置选项，通过 `menuconfig` 等工具交互式配置。

### 基本语法

```kconfig
# drivers/i2c/busses/Kconfig
config I2C_QCOM_GENI
    tristate "Qualcomm GENI I2C controller"
    depends on ARCH_QCOM || COMPILE_TEST
    select REGMAP
    help
      Say Y here to enable the Qualcomm Technologies, Inc.
      GENI based I2C controller driver.
      
      To compile this driver as a module, choose M here.
```

### 数据类型

| 类型 | 含义 | 示例 |
|------|------|------|
| `bool` | 二选一 (Y/n) | 决定是否编译 |
| `tristate` | 三选一 (Y/M/n) | 编译进内核/模块/不编译 |
| `int` | 整数 | 缓冲区大小等 |
| `hex` | 十六进制 | IO 地址等 |
| `string` | 字符串 | 默认参数等 |

### 常用关键字

```kconfig
config QCOM_SMD_RPM
    tristate "Qualcomm SMD RPM Driver"
    depends on ARCH_QCOM && OF      # 依赖条件
    select QCOM_SMD                  # 自动启用某个配置
    default y if ARCH_QCOM           # 平台匹配时默认开启
    help                              # 帮助文本
      This driver supports the SMD-RPM interface.
```

### menuconfig 的组织

```kconfig
menu "Qualcomm platform drivers"
    visible if ARCH_QCOM

source "drivers/soc/qcom/Kconfig"  # 包含子目录的 Kconfig

config QCOM_SMEM
    tristate "Qualcomm Shared Memory Manager"
    
config QCOM_SMD
    tristate "Qualcomm SMD Driver"

endmenu
```

## Makefile — 内核构建

### 基础规则

```makefile
# 单文件模块
obj-$(CONFIG_I2C_QCOM_GENI) += i2c-qcom-geni.o

# 多文件模块
obj-$(CONFIG_QCOM_SMD)      += qcom-smd.o
qcom-smd-objs               := smd-core.o smd-debug.o

# 编译进内核 (Y)
obj-y                       += core.o

# 子目录
obj-$(CONFIG_ARCH_QCOM)     += qcom/
```

### 驱动常用 Makefile 模式

```makefile
# SPDX-License-Identifier: GPL-2.0
#
# Makefile for Qualcomm platform drivers
#

obj-$(CONFIG_QCOM_SMEM)     += qcom_smem.o
obj-$(CONFIG_QCOM_SMD)      += qcom_smd.o
obj-$(CONFIG_QCOM_RMTFS)    += qcom_rmtfs.o

# 多文件驱动
qcom_smd-objs := smd_core.o smd_debug.o smd_trace.o

# 条件编译
ifdef CONFIG_DEBUG_FS
qcom_smd-objs += smd_debugfs.o
endif

# 编译标志
ccflags-y := -I$(srctree)/drivers/soc/qcom/include
```

## defconfig — 平台默认配置

每个平台都有自己的 defconfig，定义该平台的默认内核功能集：

```bash
# 查看 Qualcomm 平台的 defconfig
ls arch/arm64/configs/ | grep qcom
# 输出示例：
# defconfig
# qcom_defconfig
# qcom-linux_defconfig

# 修改 defconfig 的步骤
make qcom_defconfig           # 加载高通平台配置
make menuconfig               # 交互式修改
make savedefconfig            # 生成精简的 defconfig
cp defconfig arch/arm64/configs/qcom_defconfig  # 保存
```

### defconfig 示例片段

```text
# arch/arm64/configs/defconfig (Qualcomm 部分)
CONFIG_ARCH_QCOM=y
CONFIG_ARCH_QCOM_IPQ8074=y
CONFIG_ARCH_SDM845=y
CONFIG_ARCH_SM8250=y
CONFIG_QCOM_SMEM=y
CONFIG_QCOM_SMD=y
CONFIG_I2C_QCOM_GENI=y
CONFIG_SPI_QCOM_GENI=y
CONFIG_QCOM_EARLY_RANDOM=y
```

## 完整编译流程

```bash
# 1. 设置架构和交叉编译器
export ARCH=arm64
export CROSS_COMPILE=aarch64-linux-gnu-
export PATH=/path/to/toolchain/bin:$PATH

# 2. 加载配置
make qcom_defconfig

# 3. 编译内核
make -j$(nproc) Image.gz dtbs

# 4. 编译模块
make modules

# 5. 安装模块
make modules_install INSTALL_MOD_PATH=/path/to/rootfs

# 6. 编译独立目录下的模块（开发驱动时常用）
make -C /path/to/kernel M=$(pwd) modules
```

## 实战技能

### 1. 单模块快速编译（驱动开发者最常用）

```bash
# 只编译一个驱动模块，不编译整个内核
make -C $KERNEL_SRC M=drivers/soc/qcom modules
```

### 2. 查看配置依赖

```bash
# 查看某个配置项的具体定义和依赖
grep -r "config QCOM_SMEM" kernel/  # 查找定义位置
make menuconfig  # 图形化查看依赖树
```

### 3. 调试编译问题

```bash
# 打印完整的编译命令
make V=1  # Verbose 模式

# 查看实际使用的配置
cat .config | grep QCOM

# 检查符号是否被正确导出
nm vmlinux | grep qcom_smem
```

## 面试考点

1. **tristate 的 Y/M/N 有什么区别？** Y=编译进内核镜像，M=编译为独立模块，N=不编译
2. **Kconfig 的 depends on 和 select 的区别？** depends on 要求依赖项已启用；select 强制启用某个配置
3. **如何给 kernel 添加新的系统调用？** (很少这么做，但面试会问) 添加 syscall 号、实现函数、更新 syscall table
4. **modules.order 文件的作用？** 当多个模块顺序加载有依赖时，控制模块加载顺序
5. **dtbo.img 和 dtb.img 的区别？** dtb 是设备树 blob；dtbo 是叠加层，用于在 bootloader 层面覆盖 dts 配置
