# 性能优化 — perf / tracepoint / profiling

## 概述

性能优化是高通内核工程师的进阶技能。你需要分析系统瓶颈、优化驱动延迟、降低功耗。

> 高通场景：相机启动延迟优化、GPU 帧率调优、I2C 传输速度优化、系统功耗分析与调度优化。

## perf — Linux 性能分析器

perf 是 Linux 内核官方性能分析工具，支持硬件性能计数器、tracepoint、动态探针等。

### 安装

```bash
# 内核配置
CONFIG_PERF_EVENTS=y
CONFIG_HW_PERF_EVENTS=y

# 用户态工具
sudo apt install linux-tools-common linux-tools-generic
sudo apt install linux-tools-$(uname -r)
```

### 常用 perf 命令

```bash
# 1. perf stat — 统计程序运行时的硬件事件
perf stat sleep 5
# 输出：CPU cycles, instructions, cache misses, branch misses

# 2. perf record + perf report — 采样分析热点函数
perf record -a -g -- sleep 10      # 全系统采样10秒（-g=调用栈）
perf report                        # 显示热点函数，占比排序
perf report --sort=dso             # 按模块/驱动排序

# 3. perf top — 实时查看热点
perf top                           # 类似 top，但显示函数级别的CPU占用
perf top -p <pid>                  # 只看某个进程
perf top -k vmlinux                # 只看内核函数

# 4. perf annotate — 查看热点函数的汇编级热点
perf annotate <function_name>      # 显示函数内每条指令的采样数

# 5. perf list — 列出所有可测量的事件
perf list                          # 全部事件
perf list hw                       # 硬件事件
perf list tracepoint               # tracepoint 事件
perf list "sched:*"                # 调度相关的 tracepoint
```

### 驱动性能分析实战

```bash
# 分析 I2C 驱动传输性能
# 1. 找到 I2C 事件
perf list | grep i2c

# 2. 只跟踪 I2C 事件
perf stat -e i2c:i2c_read,i2c:i2c_write -- sleep 10

# 3. 记录 I2C 操作的时间分布
perf record -e i2c:i2c_result -ag -- sleep 10
perf report

# 4. 分析驱动函数的 CPU 使用
perf top -k vmlinux --sort symbol,dso
# 看哪个内核函数占用 CPU 最多
```

### 使用 perf 定位调度延迟

```bash
# 分析调度延迟（高通 EAS 调试常用）
perf sched record -- sleep 5       # 记录调度事件
perf sched latency                 # 显示调度延迟统计
perf sched map                     # 显示 CPU 上的进程切换图
perf sched timehist                # 调度事件时间线

# 查看特定任务的 wakeup 延迟
perf sched record -e sched:sched_wakeup,sched:sched_switch -- ./test
perf script --sched-stat
```

## tracepoint — 静态跟踪点

内核预定义的跟踪点，可以在关键函数入口/出口处收集数据。

### 常用 tracepoint

```bash
# 查看所有 tracepoint
ls /sys/kernel/debug/tracing/events/

# 子系统分类
# sched:*      — 调度器事件（唤醒、切换、阻塞）
# irq:*        — 中断事件（进入、退出）
# timer:*      — 定时器事件
# kmem:*       — 内存分配/释放
# i2c:*        — I2C 传输
# spi:*        — SPI 传输
# regulator:*  — 电压调节器
# cpufreq:*    — CPU 频率缩放
# power:*      — 电源管理
# workqueue:*  — 工作队列
# dma:*        — DMA 传输
```

### 使用 tracepoint 调试驱动

```bash
# 1. 通过 ftrace 接口使用 tracepoint
cd /sys/kernel/debug/tracing

# 开启 I2C tracepoint
echo 1 > events/i2c/i2c_read/enable
echo 1 > events/i2c/i2c_write/enable
echo 1 > tracing_on
cat trace_pipe  # 实时查看

# 关闭
echo 0 > tracing_on
echo 0 > events/i2c/i2c_read/enable

# 2. 通过 perf 使用 tracepoint
perf record -e i2c:* -ag -- sleep 5
perf script

# 3. 通过 bpftrace 使用（更灵活）
# bpftrace -e 'tracepoint:i2c:i2c_read { printf("read %s\n", comm); }'
```

### 自定义 tracepoint

你可以在自己的驱动中定义 tracepoint：

```c
#include <trace/events/qcom.h>

// 在驱动代码中创建 tracepoint 头文件
// include/trace/events/qcom.h

#if !defined(_TRACE_QCOM_H) || defined(TRACE_HEADER_MULTI_READ)
#define _TRACE_QCOM_H

#include <linux/tracepoint.h>

DECLARE_EVENT_CLASS(qcom_i2c_template,
    TP_PROTO(int id, size_t len, int ret),
    TP_ARGS(id, len, ret),
    TP_STRUCT__entry(
        __field(int, id)
        __field(size_t, len)
        __field(int, ret)
    ),
    TP_fast_assign(
        __entry->id = id;
        __entry->len = len;
        __entry->ret = ret;
    ),
    TP_printk("id=%d len=%zu ret=%d",
              __entry->id, __entry->len, __entry->ret)
);

DEFINE_EVENT(qcom_i2c_template, qcom_i2c_transfer_start,
    TP_PROTO(int id, size_t len, int ret),
    TP_ARGS(id, len, ret));

DEFINE_EVENT(qcom_i2c_template, qcom_i2c_transfer_done,
    TP_PROTO(int id, size_t len, int ret),
    TP_ARGS(id, len, ret));
#endif

/* 在驱动代码中调用 */
#include <trace/events/qcom.h>

trace_qcom_i2c_transfer_start(dev->id, msg->len, 0);
// ... 传输 ...
trace_qcom_i2c_transfer_done(dev->id, msg->len, ret);
```

## ftrace — 函数跟踪器

```bash
# function tracer — 跟踪所有内核函数调用
echo function > /sys/kernel/debug/tracing/current_tracer
echo qcom_i2c_* > /sys/kernel/debug/tracing/set_ftrace_filter
cat /sys/kernel/debug/tracing/trace_pipe

# function_graph — 显示函数调用图和执行时间
echo function_graph > /sys/kernel/debug/tracing/current_tracer
echo qcom_i2c_* > /sys/kernel/debug/tracing/set_graph_function
cat /sys/kernel/debug/tracing/trace_pipe

# 查看驱动函数的执行时间
# 输出示例：
# 2)               |  qcom_i2c_xfer() {
# 2)   0.500 us    |    qcom_i2c_wait_for_xfer();
# 2)   1.200 us    |    qcom_i2c_fill_tx_fifo();
# 2)   3.500 us    |    qcom_i2c_irq_handler();
# 2)   6.300 us    |  }  /* qcom_i2c_xfer */
```

## 性能优化方法论

```bash
# 驱动性能优化步骤

# Step 1: 测量基线
perf stat -d -r 10 -- ./your_test
# 关注：cycles, instructions-per-cycle, cache misses

# Step 2: 找到热点
perf record -a -g -- ./your_test
perf report
# 找到占用 CPU 最高的函数

# Step 3: 分析热点
perf annotate hottest_function
# 看具体哪条指令最耗时

# Step 4: 优化
# - 减少锁竞争（使用 RCU 或 percpu 变量）
# - 减少内存分配（对象复用，kmem_cache）
# - 减少中断处理时间（将耗时操作移到 threaded irq）
# - 使用 DMA 而不是 PIO
# - 批量处理而不是单次传输

# Step 5: 验证
perf stat -d -r 10 -- ./your_test
# 对比基线，确认优化效果
```

## 面试考点

1. **perf 的采样原理？** 利用 PMU 硬件计数器定期触发中断，在中断中记录当前 PC 和调用栈
2. **perf stat 和 perf record 的区别？** stat 做统计计数，record 做采样记录（可以后续分析调用栈）
3. **tracepoint 和 ftrace 的区别？** tracepoint 是代码中预定义的静态跟踪点，ftrace 是动态函数跟踪
4. **怎么判断驱动是 CPU 瓶颈还是 I/O 瓶颈？** perf stat 看 cycles 和 cache-misses 的比例
5. **函数执行时间怎么看？** function_graph tracer 会显示每个函数的执行时间
