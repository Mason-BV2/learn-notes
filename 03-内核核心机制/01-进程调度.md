# 进程调度 — CFS / EAS / 调度类

## 概述

Linux 调度器负责在多个进程之间分配 CPU 时间。从 O(1) 调度器到 CFS（完全公平调度），再到 EAS（能耗感知调度），调度器在性能和功耗之间不断进化。

> 高通工作场景：你需要理解和调试 CPU 频率缩放、调度亲和性、大小核（big.LITTLE）调度优化和功耗管理。

## CFS — 完全公平调度

CFS 由 Ingo Molnár 在 2.6.23 引入，核心思路是使用红黑树维护进程的运行权重。

### 核心数据结构

```c
// kernel/sched/sched.h
struct task_struct {
    // ...
    struct sched_entity se;      // 调度实体
    struct sched_rt_entity rt;   // 实时调度实体
    struct sched_dl_entity dl;   // Deadline 调度实体
    unsigned int policy;         // 调度策略
    int prio;                    // 静态优先级
    // ...
};

struct sched_entity {
    struct load_weight load;     // 权重
    struct rb_node run_node;     // 红黑树节点
    u64 vruntime;               // 虚拟运行时间（核心）
    u64 sum_exec_runtime;       // 实际执行时间
    // ...
};
```

### vruntime — 虚拟运行时间

```
vruntime += 实际运行时间 * (NICE_0_LOAD / se.weight)
```

- 权重越高，vruntime 增长越慢 → 获得更多 CPU 时间
- CFS 总是选择 vruntime 最小的进程运行

### 调度策略

```c
// 普通进程
#define SCHED_NORMAL    0   // 普通公平调度（CFS）
#define SCHED_BATCH     1   // 批量调度（非交互）
#define SCHED_IDLE      5   // 空闲调度

// 实时进程
#define SCHED_FIFO      2   // 先入先出
#define SCHED_RR        3   // 时间片轮转

// Deadline 调度
#define SCHED_DEADLINE  6   // 最早截止时间优先
```

### 调度类（Scheduling Class）优先级

```
stop_sched_class     → 停止任务（最高优先级）
dl_sched_class       → Deadline
rt_sched_class       → 实时
fair_sched_class     → CFS（普通进程）
idle_sched_class     → 空闲（最低优先级）
```

## EAS — 能耗感知调度

EAS 是 ARM big.LITTLE 平台上的关键调度技术，在高通 Snapdragon 芯片上广泛使用。

### 核心概念

```c
// kernel/sched/energy.c — Energy Model
struct em_perf_domain {
    struct em_perf_state *table;  // 每个频率的性能/功耗表
    int nr_perf_states;           // 频率状态数
    unsigned long flags;          // 标记
};

struct em_perf_state {
    unsigned long frequency;    // kHz
    unsigned long power;        // mW
    unsigned long cost;         // 性能-功耗比
    unsigned long max;          // 最大性能
};
```

### EAS 工作流程

```
任务 wakeup → select_task_rq_fair()
  → find_energy_efficient_cpu()  ← EAS 核心函数
    → 计算每个 CPU 的能耗
    → 选择最节能的 CPU
  → 如果 EAS 找不到更好的，fallback 到传统 wake_affine()
```

### Energy Model 初始化

```c
// 在 DTS 中定义 CPU 功耗
// arch/arm64/boot/dts/qcom/sdm845.dtsi
cpu0: cpu@0 {
    compatible = "qcom,kryo385";
    reg = <0x0 0x0>;
    capacity-dmips-mhz = <1024>;  // 性能容量
    dynamic-power-coefficient = <290>;  // 功耗系数
};

// 调度域中开启 EAS
// 需要支持 SD_ASYM_CPUCAPACITY 标记
```

## Load Tracking — PELT

CFS 使用 PELT（Per-Entity Load Tracking）跟踪每个调度实体的负载：

```c
/*
 * 每个调度实体维护过去负载的衰减总和
 *
 * 负载计算 = 0.5^t * load(t) + ...
 * 半衰期 = 32ms
 *
 * 最近时间权重高，越旧的负载权重越低
 */
struct sched_avg {
    u64 last_update_time;       // 上次更新时间
    u64 load_sum;               // 负载总和
    u64 runnable_sum;           // 可运行总和
    u32 util_sum;               // 利用率总和
    unsigned long load_avg;     // 平均负载
    unsigned long runnable_avg; // 平均可运行
    unsigned long util_avg;     // 平均利用率
};
```

## cpufreq — CPU 频率缩放

```c
// drivers/cpufreq/cpufreq.c

// cpufreq 核心结构
struct cpufreq_policy {
    struct cpufreq_frequency_table *freq_table;
    struct cpufreq_governor *governor;
    unsigned int min;    // 最小频率 (kHz)
    unsigned int max;    // 最大频率 (kHz)
    unsigned int cur;    // 当前频率 (kHz)
    // ...
};

// cpufreq governor（策略）
// performance  → 固定最高频率
// powersave    → 固定最低频率
// userspace    → 用户控制
// ondemand     → 按需调频（旧）
// conservative → 保守调频
// schedutil    → 基于调度器 util 调频 ← 高通常用
```

### schedutil — EAS 的核心 governor

```
schedutil 直接从 PELT util_avg 读取 CPU 负载
→ 计算目标频率
→ 通过 cpufreq 驱动设置频率

这比旧的 ondemand 更快（无采样延迟）
```

```c
// kernel/sched/cpufreq_schedutil.c
static unsigned int get_next_freq(struct sugov_policy *sg_policy,
                                   unsigned long util, unsigned long max)
{
    // util 来自 PELT util_avg
    // 目标频率 = util/max * max_freq
    return map_util_freq(util, freq, max);
}
```

## Cgroup — 资源控制组

```c
// 使用 cgroup 限制进程组对 CPU 的访问
// /sys/fs/cgroup/cpu/

// 创建控制组
mkdir /sys/fs/cgroup/cpu/qcom_group

// 设置 CPU 份额（默认 1024）
echo 512 > /sys/fs/cgroup/cpu/qcom_group/cpu.shares

// 把进程加入控制组
echo 1234 > /sys/fs/cgroup/cpu/qcom_group/tasks
```

## 实操练习

### 1. 查看当前调度器和调度策略

```bash
# 查看当前使用的调度器
cat /sys/kernel/debug/sched/preempt

# 查看 CPU 信息
lscpu

# 查看进程的调度策略
chrt -p 1

# 查看 cpufreq governor
cat /sys/devices/system/cpu/cpu0/cpufreq/scaling_governor
```

### 2. 查看 EAS 状态

```bash
# 检查 EAS 是否启用（需要内核支持）
cat /sys/kernel/debug/sched/energy_cpu

# 查看 CPU capacity（EAS 需要非对称 capacity）
cat /sys/devices/system/cpu/cpu*/cpu_capacity
```

## 面试考点

1. **CFS 如何保证公平性？** 通过 vruntime 红黑树跟踪每个进程的虚拟运行时间，选择最小值运行
2. **EAS 和 CFS 的关系？** EAS 是 CFS 之上的优化，在唤醒时决定将任务放在哪个 CPU 上运行，以最小化能耗
3. **PELT 如何计算负载？** 使用半衰期为 32ms 的指数衰减算法，越近的负载权重越高
4. **schedutil 比 ondemand 好在哪里？** schedutil 直接从调度器获取负载（无采样延迟），能更快响应负载变化
5. **big.LITTLE 迁移策略有哪些？** 全局任务调度（GTS）、CPU 集群迁移、In-keepers 等
