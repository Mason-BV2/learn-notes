# 并发与同步 — spinlock / mutex / RCU / lockdep

## 概述

并发是内核开发最核心、最容易出bug的领域。高通平台多核场景下，竞态条件（race condition）是驱动调试中最常见的问题。

> 面试必考，驱动开发天天用。不理解并发安全，写不出能合入主线的驱动。

## 并发问题的来源

```c
// 内核中的并发来源
// 1. 中断处理 → 抢占当前执行的代码
// 2. 软中断/tasklet → 在软中断上下文执行
// 3. 内核抢占 → CONFIG_PREEMPT 导致进程被抢走CPU
// 4. 多核SMP → 两个核同时执行同一段代码
// 5. 锁顺序 → 不同路径以不同顺序拿锁 → 死锁

// 保护什么？
// 1. 全局变量 / 静态变量
// 2. 设备结构体中的共享数据
// 3. 硬件寄存器（寄存器并发访问）
// 4. 链表 / 哈希表等数据结构
```

## 互斥机制层次

```
硬件层
  └── 内存屏障 (DMB/DSB/ISB)
  └── 原子操作 (atomic_t / atomic64_t)

软件层 — 忙等待
  └── spinlock (自旋锁)
  └── rwlock (读写自旋锁)
  └── raw_spinlock (原始自旋锁，RT内核中唯一真正的自旋锁)

软件层 — 可睡眠
  └── mutex (互斥锁)
  └── rt_mutex (实时互斥锁，PI协议)
  └── semaphore (信号量，不推荐)
  └── completion (完成量)

软件层 — 无锁
  └── RCU (Read-Copy-Update)
  └── 原子操作 + 内存屏障

调试层
  └── lockdep (锁依赖性验证)
  └── kmemleak
  └── KCSAN (数据竞争检测)
```

## spinlock — 自旋锁

spinlock 是内核中最基本的锁，"自旋"的意思是拿不到锁就原地循环等待。

### 基本用法

```c
#include <linux/spinlock.h>

// 定义锁
spinlock_t my_lock;
DEFINE_SPINLOCK(my_lock);          // 静态初始化
// spin_lock_init(&my_lock);      // 动态初始化

// 基本使用
spin_lock(&my_lock);               // 获取锁
// ... 临界区（不能sleep！不能调用可能sleep的函数！）
spin_unlock(&my_lock);             // 释放锁

// 关中断 + 获取锁（在中断处理中保护共享数据时必需）
unsigned long flags;
spin_lock_irqsave(&my_lock, flags);  // 保存本地中断状态 + 关中断 + 拿锁
// ... 临界区
spin_unlock_irqrestore(&my_lock, flags);  // 释放锁 + 恢复中断
```

### spinlock 的变体

```c
// spinlock
spin_lock() / spin_unlock()
// 最简单的自旋锁，临界区不能sleep

// spinlock + 关本地中断
spin_lock_irq() / spin_unlock_irq()  
// 用于确定中断是打开的进程上下文
// spin_lock_irqsave() / spin_unlock_irqrestore()
// 保存中断状态的版本（推荐用这个）

// spinlock + 底半部禁用
spin_lock_bh() / spin_unlock_bh()
// 用于保护被 softirq/tasklet 和进程上下文共享的数据

// 读写自旋锁
rwlock_t my_rwlock;
read_lock(&my_rwlock);     // 多个读者可以同时持有
read_unlock(&my_rwlock);
write_lock(&my_rwlock);    // 写者互斥
write_unlock(&my_rwlock);
```

### spinlock 使用规则

```c
// 规则1：临界区必须非常短（微秒级）
// 规则2：临界区内不能 sleep（不能调用 kmalloc(GFP_KERNEL)、copy_from_user 等）
// 规则3：必须确保所有路径的锁顺序一致（否则死锁）
// 规则4：同一个函数不能递归拿同一个 spinlock（spinlock 不是递归锁）

// 错误示例 — 临界区内调用了可能 sleep 的函数
spin_lock(&lock);
data = kmalloc(sizeof(*data), GFP_KERNEL);  // BUG! 可能睡眠
spin_unlock(&lock);
```

### 高通常见场景

```c
// 高通驱动中 spinlock 保护寄存器访问
struct qcom_geni_i2c_dev {
    // ...
    spinlock_t lock;       // 保护硬件寄存器访问
    void __iomem *base;    // MMIO 基地址
    int irq;
};

static irqreturn_t qcom_geni_i2c_irq(int irq, void *dev)
{
    struct qcom_geni_i2c_dev *dev = dev;
    unsigned long flags;
    
    spin_lock_irqsave(&dev->lock, flags);
    // 读取/写入硬件寄存器
    writel_relaxed(val, dev->base + REG_STATUS);
    spin_unlock_irqrestore(&dev->lock, flags);
    
    return IRQ_HANDLED;
}
```

## mutex — 互斥锁

mutex 是可睡眠的互斥锁，用于进程上下文。

```c
#include <linux/mutex.h>

// 定义mutex
struct mutex my_mutex;
DEFINE_MUTEX(my_mutex);           // 静态初始化
// mutex_init(&my_mutex);        // 动态初始化

// 使用
mutex_lock(&my_mutex);            // 获取锁（可能睡眠）
// ... 临界区（可以sleep！）
mutex_unlock(&my_mutex);          // 释放锁

// 非阻塞尝试
if (mutex_trylock(&my_mutex)) {
    // 拿到了锁
    mutex_unlock(&my_mutex);
} else {
    // 没拿到，做别的事
}

// 可中断版本（推荐）
mutex_lock_interruptible(&my_mutex);  // 如果收到信号返回 -ERESTARTSYS
// ... 临界区
mutex_unlock(&my_mutex);
```

### spinlock vs mutex 选择表

| 场景 | 用 spinlock | 用 mutex |
|------|------------|---------|
| 中断上下文 | ✅ 必须 | ❌ 不能 |
| 临界区很小（几微秒） | ✅ | ✅ 也可以 |
| 临界区很大（毫秒级） | ❌ 浪费CPU | ✅ |
| 需要睡眠（如分配内存） | ❌ | ✅ |
| 保护 MMIO 寄存器 | ✅ 推荐 | ❌ |
| 进程上下文 + 大操作 | ❌ | ✅ 推荐 |

## RCU — Read-Copy-Update

RCU 是 Linux 内核中最强大的无锁同步机制，读操作完全没有锁开销。

```c
// RCU 的核心思想：
// 读路径 → 不需要任何锁（没有原子操作、没有内存屏障）
// 写路径 → 复制数据 → 修改 → 发布新指针 → 等待所有读者完成 → 释放旧数据

#include <linux/rcupdate.h>

// 被 RCU 保护的数据（必须是指针）
struct my_data {
    int value;
};
struct my_data __rcu *gptr;

// 读路径（完全没有锁）
rcu_read_lock();                        // 标记 RCU 读临界区
struct my_data *p = rcu_dereference(gptr);  // 读取安全指针
if (p)
    val = p->value;                     // 使用数据
rcu_read_unlock();                      // 退出 RCU 读临界区

// 写路径（复制 + 替换 + 等待）
struct my_data *new = kmalloc(sizeof(*new), GFP_KERNEL);
new->value = 42;
struct my_data *old = rcu_replace_pointer(gptr, new, lockdep_is_held(&some_lock));
synchronize_rcu();                      // 等待所有读者完成
kfree(old);                             // 安全释放旧数据
```

### RCU 使用场景

```c
// 场景：路由表、配置数据、状态表
// 读多写极少、读路径必须极快、能接受写路径稍慢

// 内核中 RCU 的实际应用
// - 路由表 (net/ipv4/route.c)
// - 文件系统 dentry 缓存
// - 进程 ID 查找
// - 模块列表
```

## lockdep — 锁依赖性验证

lockdep 是内核内置的锁调试工具，能自动检测死锁风险。

```c
// 内核配置
CONFIG_LOCKDEP=y           // 必须开启
CONFIG_PROVE_LOCKING=y     // 推荐
CONFIG_DEBUG_LOCK_ALLOC=y  // 推荐

// lockdep 能检测的问题
// 1. 死锁（ABBA 问题）
// 2. 锁顺序不一致
// 3. 锁的递归使用
// 4. 在中断上下文中使用可睡眠的锁
// 5. 遗漏了锁的初始化

// lockdep 输出示例
// 
// ============================================
// WARNING: possible recursive locking detected
// --------------------------------------------
// insmod/1234 is trying to acquire lock:
//  (&dev->lock){....}, at: [<...>] my_function+0x10/0x50
//
// but task is already holding lock:
//  (&dev->lock){....}, at: [<...>] my_function+0x8/0x50
//
// 说明：同一个函数递归拿了同一个锁
```

### 锁使用最佳实践

```c
// 1. 锁顺序必须全局一致
// 例如：先拿 dev->lock，再拿 dev->sub_lock
// 所有代码路径都必须按这个顺序

// 2. 使用 lockdep 注解
/* When lockdep is enabled, spin_lock_init starts annotation */
spin_lock_init(&dev->lock);  // 告诉 lockdep 这个锁的存在

// 3. 标注锁的持有者
struct qcom_device {
    spinlock_t lock;          // 保护: hw_regs, irq_status
    // ...
};

// 4. 使用 sparse 静态检查
// __acquires / __releases 标注
static void my_func(struct qcom_device *dev)
    __acquires(&dev->lock)
{
    spin_lock(&dev->lock);
}

static void my_func_end(struct qcom_device *dev)
    __releases(&dev->lock)
{
    spin_unlock(&dev->lock);
}
```

## 死锁场景与避免

```c
// 经典 ABBA 死锁
// Thread A: 拿锁A → 试图拿锁B
// Thread B: 拿锁B → 试图拿锁A

// 解决方案：全局锁顺序
// 所有线程先拿锁A再拿锁B

// 自死锁 — spinlock 不能递归
// BUG!
spin_lock(&lock);
// ...
spin_lock(&lock);  // 自旋等待自己释放锁 → 死锁

// 解决方案：区分 spinlock 和 mutex
// spinlock 不递归
// mutex 可以递归吗？不行！mutex 也不允许递归
// 需要递归锁？改用完成量或自己实现计数

// 中断上下文死锁
// 进程上下文持有锁 → 中断发生 → 中断处理也拿同一个锁
spin_lock(&lock);           // 进程拿了锁
// ← 中断来了
spin_lock(&lock);           // 中断也拿锁 → 死锁

// 解决方案：使用 spin_lock_irqsave
spin_lock_irqsave(&lock, flags);  // 关中断 + 拿锁
```

### 练习2：编写一个有并发bug的模块并修复

```c
// 有 bug 的模块 — 没有锁保护共享数据
static int shared_counter = 0;

static ssize_t buggy_write(struct file *filp, const char __user *buf,
                            size_t len, loff_t *off)
{
    int local = shared_counter;  // 可能被中断抢占
    local++;                     // 另一个CPU可能也在执行
    shared_counter = local;      // 数据不一致！
    return len;
}

// 修复版本
static spinlock_t counter_lock;
static int shared_counter = 0;

static ssize_t fixed_write(struct file *filp, const char __user *buf,
                            size_t len, loff_t *off)
{
    unsigned long flags;
    spin_lock_irqsave(&counter_lock, flags);
    shared_counter++;
    spin_unlock_irqrestore(&counter_lock, flags);
    return len;
}
```

## 面试考点

1. **spinlock 和 mutex 的根本区别？** spinlock 忙等待（不能sleep），mutex 可睡眠（需要进程上下文）
2. **什么时候必须用 spin_lock_irqsave？** 当锁可能被中断上下文和进程上下文同时访问时
3. **RCU 为什么读路径更快？** 读路径没有原子操作、没有内存屏障、没有锁的获取/释放
4. **如何检测死锁？** lockdep 自动检测，看 locking self-test 输出
5. **percpu 变量如何避免锁？** 每个CPU有自己的副本，不需要互斥访问
